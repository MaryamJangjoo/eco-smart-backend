# چک‌لیست عملیاتی مهاجرت معماری پلتفرم (Migration Checklist)

این سند وضعیت پیشرفت لحظه‌ای انتقال کدهای پروژه را از ساختار سنتی (AS-IS) به ساختار تفکیک‌شده پلتفرم (TO-BE) ردیابی می‌کند. هر گام باید پس از تست موفقیت‌آمیز تیک بخورد.

---

## 🏁 Phase 0: تثبیت زون راه‌اندازی (Application Layer)
هدف: جابه‌جایی فایل‌های ریشه سیستم بدون تغییر در منطق کدهای فنی یا بیزنسی.

- [ ] **۰.۱ ایجاد دایرکتوری مقصد:** ساخت پوشه `src/application/`
- [ ] **۰.۲ جابه‌جایی امن فایل اصلی:** اجرای `git mv src/main.ts src/application/main.ts`
- [ ] **۰.۳ جابه‌جایی امن ماژول ریشه:** اجرای `git mv src/app.module.ts src/application/app.module.ts`
- [ ] **۰.۴ اصلاح آدرس‌دهی ریشه:** اصلاح `import { AppModule } from './app.module'` درون فایل `main.ts`
- [ ] **۰.۵ اصلاح آدرس‌دهی ماژول‌ها:** به‌روزرسانی مسیر ماژول‌های قدیمی در `app.module.ts` (تغییر `./modules/` به `../modules/`)
- [ ] **۰.۶ ارزیابی کامپایلر (TypeScript):** اجرای `npm run build` بدون پدیدار شدن خطای گم‌شدن مسیر (Missing Imports)
- [ ] **۰.۷ ارزیابی ارکستراسیون (Docker):** اجرای `docker compose up -d --build` و تایید بالا آمدن بک‌آند بدون کرش تکراری

---

## 🔐 Phase 1: تفکیک هاب هویتی (Platform Identity)
هدف: انتقال مستقل سرویس‌های پایه کاربران و احراز هویت.

- [ ] **۱.۱ ایجاد دایرکتوری:** ساخت پوشه `src/platform/identity/`
- [ ] **۱.۲ انتقال Auth:** جابه‌جایی ماژول احراز هویت با `git mv` به مسیر `src/platform/identity/auth/`
- [ ] **۱.۳ انتقال Users:** جابه‌جایی ماژول کاربران با `git mv` به مسیر `src/platform/identity/users/`
- [ ] **۱.۴ اصلاح رفرنس‌ها:** به‌روزرسانی تمام اینپورت‌های بیرونی که از این دو ماژول استفاده می‌کردند.
- [ ] **۱.۵ تست سلامت:** اجرای Build پروژه و تایید کارکرد درست مکانیزم ورود (Login) و صدور JWT.

---

## 🔌 Phase 2: مجزاسازی درایور سخت‌افزار (Infrastructure Layer)
هدف: پاک‌سازی کدهای بیزنسی از منطق فریم‌های خام شبکه.

- [ ] **۲.۱ ایجاد دایرکتوری:** ساخت پوشه `src/infrastructure/mybus/`
- [ ] **۲.۲ انتقال کدهای پارسر:** جابه‌جایی توابع مبدل بایت و محاسبات CRC به لایه زیرساخت.
- [ ] **۲.۳ تست یکپارچگی:** قطع وابستگی مستقیم ماژول دیوایس به فریم‌های خام و راه‌اندازی Interfaceهای انتزاعی.
در ساختار فعلی شما:

backend-api/src
├── app.module.ts
├── common/
├── main.ts
└── modules/
    ├── auth/
    ├── devices/
    ├── health/
    └── users/

پس انتقال app.module.ts و main.ts به application/ درست است، اما باید حواسمان باشد که در این مرحله modules را جابه‌جا نمی‌کنیم. فقط مسیر import آنها در app.module.ts تغییر می‌کند.

یعنی بعد از Phase 0 انتظار داریم:

backend-api/src
├── application/
│   ├── app.module.ts
│   └── main.ts
│
├── common/
│
└── modules/
    ├── auth/
    ├── devices/
    ├── health/
    └── users/

این وضعیت کاملاً معتبر است.
