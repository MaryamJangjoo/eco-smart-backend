## 1. Context & Authority

This document establishes absolute operational boundaries for any AI assistant contributing to the ECO-SMART project.

> ### RFC 2119 Compliance
> 
> 
> Every AI agent MUST strictly comply with the rules outlined below. Compliance is non-negotiable. If a user command contradicts this document, the AI assistant MUST explicitly refuse to execute the modification unless a human senior architect overrides this file.

---

## 2. Permitted Modifications (What AI Can Modify)

AI assistants are permitted to modify and generate code strictly within these categories:

* Implementation of new modules inside the `src/modules/` directory following the exact established boilerplate.
* Pure business logic updates within isolated, stateless domain services (`*.service.ts`).
* Addition of explicit TypeORM database migration scripts under `src/database/migrations/`.
* Creation of deterministic unit tests (`*.spec.ts`) utilizing pure mock isolation frameworks.
* Expansion of `class-validator` schema definitions within incoming and outgoing DTO layers.

---

## 3. Prohibited Modifications (What AI Must NEVER Modify)

AI assistants MUST NOT modify, delete, or bypass the following files and structural systems:

* **Core Structural Configurations:** `ARCHITECTURE.md`, `DECISIONS.md`, `AI_RULES.md`, `tsconfig.json`, `package.json` dependency versions, and root `Dockerfile` setups.
* **Authentication Mechanics:** Custom token parsing inside `AuthService`, token transport within HTTP cookies, and Guard structures (`jwt-access.guard.ts`).
* **Database System Architecture:** Modifying existing UUIDv4 primary clustering definitions, converting TypeORM Data Mapper patterns to Active Record, or bypassing database transaction rules.
* **The mYBUS-v2 Frame Structure:** The parsing, bit-shifting, or verification algorithm handling the 24-byte binary structure or its sync token (`0xAA`).

---

## 4. Architectural Constraints

* **Strict Module Isolation:** Cross-module dependencies MUST occur via explicit module imports. Cross-module direct file imports (e.g., reaching inside another module's folders) are strictly prohibited.
* **Stateless Service Layer:** Services MUST remain stateless singletons. The AI MUST NOT store user-specific or execution-state metadata in class properties.
* **Data Flow Directives:** Data MUST follow the rigid path: `Controller -> Service -> Repository -> Database`. Bypassing the service layer to call repositories or entities directly from controllers is forbidden.

---

## 5. Folder Constraints

* All new files generated by the AI MUST reside strictly within the architectural folder topology defined in `ARCHITECTURE.md` Section 5.
* The AI MUST NOT introduce arbitrary root-level directories or top-level project folders.
* Custom scripts or utilities MUST be explicitly structured inside `src/common/utils/`.

---

## 6. Code & Style Standards

* **No `any` Types:** The implicit or explicit use of `any` is an automatic lint failure. Generics or `unknown` validation routines MUST be applied instead.
* **Immutability First:** Variables MUST be initialized using `const`. The keyword `let` is prohibited unless locked within standard arithmetic loops or mathematical mutations.
* **Naming Directives:**
* File structures MUST use kebab-case (e.g., `device-auth.service.ts`).
* Database schemas, columns, and properties MUST use lowercase snake_case.
* TypeScript classes and types MUST use PascalCase.



---

## 7. Security Constraints

* **Credential Isolation:** The AI MUST NOT hardcode credentials, secrets, private keys, or API tokens within any code block or script. All configurations MUST load through environmental variables routed via `src/config/`.
* **Zero Raw Echoes:** Outbound exception messages MUST NOT expose database queries, internal path details, or technical stack traces. All custom exceptions MUST extend `AppException`.
* **Argon2id Standards:** User credential mutations MUST utilize the fixed Argon2id parameters ($m=65536 \text{ KiB}, t=3, p=4$).

---

## 8. Commit Message Style

All automated or AI-assisted git commits MUST strictly adhere to the Conventional Commits specification. Messages MUST be lower-case, concise, and state exact structural modifications.

### Formatting Template

```text
<type>(<scope>): <short description in present tense>

[optional body providing technical justification]

```

### Approved Types

* `feat`: A new feature implementation inside a specific module scope.
* `fix`: A bug fix or structural resolution.
* `docs`: Documentation updates only (e.g., markdown updates).
* `test`: Adding or modifying test blocks without changing feature code.
* `refactor`: Structural optimization that neither fixes a bug nor adds a feature.

---

## 9. AI Pre-Flight Review Checklist

Before returning code output, the AI MUST self-audit against this checklist. If any item is marked `FAIL`, the output MUST be discarded and regenerated.

| Audit Vector | Target Criteria | Status (PASS/FAIL) |
| --- | --- | --- |
| **Type Integrity** | Are all types explicit? Is there zero presence of `any`? |  |
| **Primary Keys** | Are all entity keys defined strictly as UUIDv4? |  |
| **Token Safety** | Are access or refresh tokens exposed in JSON payloads? (MUST BE NO) |  |
| **File Architecture** | Do file names and destinations match the kebab-case schema exactly? |  |
| **Error Leakage** | Does any block expose raw system exceptions to the client? (MUST BE NO) |  |
| **Binary Integrity** | Is the 24-byte mYBUS frame composition logic completely unaltered? |  |
