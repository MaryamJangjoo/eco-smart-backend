# برنامه مهاجرت تدریجی به معماری پلتفرم (Migration Plan)

این سند مراحل، ترتیبات و الزامات انتقال ساختار فعلی پروژه `ECO-SMART` به معماری هدف (Modular Monolith) را تبیین می‌کند. 

## 🚨 قوانین حاکم بر فاز مهاجرت (Migration Guardrails)
1. **قانون بازسازی امن:** بعد از اجرای هر گام، پروژه باید بدون خطا کامپایل (`npm run build`) شده و کانتینر داکر بک‌آند (`ecosmart_backend_api`) بدون کرش بالا بیاید.
2. **حفظ تاریخچه گیت:** برای جابه‌جایی فایل‌ها صرفاً از دستور `git mv` استفاده می‌شود تا History و Blame فایل‌ها از بین نرود.
3. **توقف توسعه ویژگی جدید:** در طول اجرای هر گام، هیچ فیچر بیزنسی جدیدی روی ماژول در حال انتقال نوشته نمی‌شود.

---

## 🗺️ فازهای عملیاتی مهاجرت

### 🏁 گام صفر: آماده‌سازی زون راه‌اندازی (Application & Core)
پیش از دست زدن به کدهای بیزنسی، باید ساختار فنی و بوت‌استرپ پروژه را مستقر کنیم.
* [ ] ایجاد پوشه `src/application` و انتقال فایلهای `main.ts`، `app.module.ts` و فایل‌های پیکربندی گلوبال به درون آن.
* [ ] ایجاد پوشه `src/core` و ساخت زیرپوشه‌های `database`، `logger` و `shared`.
* [ ] انتقال کانفیگ‌های اصلی TypeORM و اتصالات دیتابیس به `core/database`.
* [ ] **تست صحت گام:** اجرای `docker-compose restart` و بررسی لاگ‌ها برای اطمینان از صحت آدرس‌دهی‌های فایل ریشه.

### 🔐 گام اول: مجزا‌سازی هاب هویتی (Platform Identity)
انتقال موجودیت‌ها و سرویس‌های پایه که تمام محصولات آینده به آن‌ها وابسته هستند.
* [ ] ایجاد دایرکتوری `src/platform/identity` و زیرپوشه‌های `users` و `auth`.
* [ ] انتقال انتیتی اصلاح‌شده‌ی `User` (`user.entity.ts`) به `platform/identity/users/`.
* [ ] انتقال ماژول احراز هویت، استراتژی‌های JWT و رفرش توکن به `platform/identity/auth/`.
* [ ] اصلاح مسیرهای Import در کل پروژه (Fixing Imports).
* [ ] **تست صحت گام:** تست صحت صدور توکن و بالا آمدن Swagger.

### 🔌 گام دوم: پاک‌سازی لایه زیرساخت (Infrastructure Boundary)
جدا کردن کدهای ارتباطی سخت‌افزار از منطق ویژگی‌های محصول.
* [ ] ایجاد دایرکتوری `src/infrastructure/mybus/`.
* [ ] انتقال کدهای مرتبط با پارس کردن بایت‌ها، محاسبات CRC و پروتکل شبکه `mYbus` به این پوشه.
* [ ] **تصفیه بیزنسی:** حذف هرگونه ارجاع به مفهوم `Device` یا `Room` از داخل پوشه `mybus` و بازنویسی آن به صورت دریافت/ارسالِ فریمِ خام.
* [ ] انتقال ماژول‌های جانبی مثل `mail` (اتصال به Mailhog) به `src/infrastructure/mail/`.

### 🏢 گام سوم: انتقال ماژول‌های اختصاصی محصول (ECO-SMART Products)
کپسوله کردن ویژگی‌های محصول هوشمندسازی در زون اختصاصی خودش.
* [ ] ایجاد ساختار تجمیعی `src/products/eco-smart/modules/`.
* [ ] انتقال پوشه‌های `devices`، `telemetry` و `automation` به درون پوشه `modules`.
* [ ] تعریف یک ماژول واحد به نام `EcoSmartModule` در ریشه این پوشه جهت بارگذاری تمام زیرماژول‌ها.
* [ ] متصل کردن `EcoSmartModule` به ماژول ریشه اپلیکیشن (`app.module.ts`).

### 📡 گام چهارم: پیاده‌سازی رویدادهای پلتفرم (Platform Event Bus)
قطع وابستگی مستقیم (Tight Coupling) ماژول‌ها به یکدیگر.
* [ ] نصب و کانفیگ `@nestjs/event-emitter` در لایه `src/platform/event-bus`.
* [ ] تعریف اینترفیس‌ها و کلاس‌های رویداد استاندارد (مانند `UserCreatedEvent` یا `TelemetryReceivedEvent`).
* [ ] جایگزینی تزریق مستقیم سرویس‌ها (Direct Injection) با `EventEmitter2.emit()`.
